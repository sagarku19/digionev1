## 4. RLS gaps

### 4.1 The headline

`security-model.md` and `supabase/exports/RLS-POLICIES.md` describe a complete policy set across 59 tables, designed and validated in a throwaway Postgres-18 cluster on 2026-06-02. **None of it is live.** Live state:

- `public.public_images` — RLS enabled ✅
- Every other public table — RLS disabled ❌
- `supabase/migrations/20260602000000_rls_policies.sql` exists in the repo but Supabase's `list_migrations` does not show it (and the `rls-status.txt` snapshot confirms `f` everywhere except `public_images`).

**Operational impact:** with only the anon key, an attacker can `SELECT *` from every table — orders, balances, ledger, KYC bank fields, license keys, user PII, all sites, all products, all everything. They can also `INSERT/UPDATE/DELETE` rows wherever schema constraints don't catch it. The "service-role-writes-only" rule is currently enforced by *convention in the API code*, not by the database.

This is by far the most important finding. Treat it as a security incident, not a tech-debt item.

### 4.2 Sensitive-column exposures (compounding #4.1)

From `get_advisors` security:

| Table | Sensitive column |
|---|---|
| `creator_payout_methods` | `account_number` |
| `product_licenses` | `license_key` |
| `conversion_events` | `session_id` |
| `product_view_events` | `session_id` |
| `site_page_views` | `session_id` |

`creator_kyc` likely also exposes `bank_account_enc`, `aadhaar_last4`, etc. but the advisor only matches obvious column names.

### 4.3 `SECURITY DEFINER` functions callable by anon/authenticated

| Function | Risk |
|---|---|
| `handle_new_user()` | Intended as trigger on `auth.users`. Anon can POST `/rest/v1/rpc/handle_new_user` and run it standalone. |
| `handle_user_email_confirmed()` | Same. |
| `sum_bucket_bytes_for_prefix(p_bucket_id, p_prefix)` | New (2026-06-03). Used by `/api/upload` quota check. Anon can call it and probe per-creator storage totals by guessing prefixes. |

Fix: `REVOKE EXECUTE ON FUNCTION … FROM anon, authenticated;` (and from `PUBLIC` to be safe). Triggers don't need EXECUTE perms to fire.

### 4.4 Storage / auth-level findings

- **`public-asset` bucket SELECT policy lets anon list every file** (advisor `public_bucket_allows_listing`). Public buckets don't need a broad SELECT — object URLs work without it.
- **Auth leaked-password protection disabled** — turn on HaveIBeenPwned check in Supabase auth settings.

### 5.1 Critical bug — `/api/payouts/request` uses the wrong creator_id

`creator_kyc.creator_id`, `creator_balances.creator_id`, `creator_payouts.creator_id` all reference **`profiles.id`** (FKs in `database.types.ts:254-262`, `creator_kyc`, `creator_payouts`). Per `RLS-POLICIES.md` core model: `auth.uid() = users.id = profiles.user_id`, then `profiles.id` is the `creator_id` used everywhere.

`app/api/payouts/request/route.ts`:
- `:27` `.eq('creator_id', user.id)` on `creator_kyc`
- `:38` `.eq('creator_id', user.id)` on `creator_balances`
- `:60` `.eq('creator_id', user.id)` on update
- `:72` `creator_id: user.id` on `creator_payouts` insert

`user.id` is **`auth.users.id`** (from `supabase.auth.getUser()`), not `profiles.id`. They are different UUIDs. The query at `:24` will return `null`, the route returns 403 "KYC must be verified", and a creator can never withdraw — even with a verified KYC row and a positive balance.

The dashboard `useEarnings` hook gets this right (`useEarnings.ts:16,19-21` calls `getCreatorProfileId()` first). Apply the same fix to the API route — resolve `user.id` → `profile.id` via `lib/auth-resolve.ts` or `lib/getCreatorProfileId.ts`-server equivalent.

This bug is masked because (a) `creator_balances` has 0 rows in prod and (b) the dev who tested probably had `user.id === profile.id` in some seeded fixture or simply hit 403 and moved on.

### 5.2 Defensive fallbacks that are now dead code

- `/api/checkout/create:89-96` retries insert without `creator_id` if it errors — column exists in types and DB, fallback never executes.
- `/api/checkout/create:111-115` and `/api/webhook/cashfree:62-66` retry without `payment_verified_at` — same. Column exists.

Not a bug, but they obscure the real shape and silence type errors with `as any`. Clean up.

### 5.3 Webhook still reads `creator_id` from `order.metadata`

`/api/webhook/cashfree:70`: `const creatorProfileId = (order.metadata as any)?.creator_profile_id ?? null;`. The `orders` table has a real `creator_id` column (and a FK to `profiles`). Read `order.creator_id` directly; keep metadata as a backup for legacy rows or drop it entirely.

## 7. Foreign-key & constraint gaps

### 7.1 Joins-in-code without a FK in DB

Skimmed the main hooks. Most joins go through PK-FK pairs that exist. Exceptions worth knowing:

- `useAnalytics`, `useCustomers` do client-side joins by re-issuing `order_items.product_id IN (…)` then `products.id IN (…)`. No FK is missing (it exists as `fk_oi_product`) but the FK is **unindexed** — see §3.1.
- `useLibrary.ts` joins `orders → order_items → products` by IN-list. Same shape — fine but slow at scale.
- `/api/webhook/cashfree` looks up `order.metadata.creator_profile_id` as a freeform JSON field. There's no DB-level guarantee that it points at a real profile. Switch to `orders.creator_id` (which has the FK).

### 7.2 Constraints worth adding

- `orders.creator_id` is nullable but the FK exists. Backfill from `metadata.creator_profile_id` (or via `order_items → products.creator_id`) and `SET NOT NULL`. Removes the fallback paths in §5.2.
- `orders.gateway_order_id` is nullable in types. It's our webhook lookup key — should be `NOT NULL UNIQUE` for paid orders. (Free orders never get one.) Consider a partial unique index: `UNIQUE (gateway_order_id) WHERE gateway_order_id IS NOT NULL`.
- `transaction_ledger.record_hash` — should be `UNIQUE` to make double-credit physically impossible. Today the only protection is the "idempotency on order.status" check in §5.D, which is logical, not structural.

### 7.3 Missing FKs

Spot check: `linkinbio_items.product_id` does have FK `fk_linkinbio_items_product` per advisor (just unindexed). `user_product_access.product_id` — confirm via schema; `/api/deliverables/[productId]/route.ts:53` reads on `user_id + product_id` and expects integrity.
